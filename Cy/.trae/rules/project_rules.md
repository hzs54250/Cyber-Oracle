# 项目规则

## 项目目标
Vibe Coding 指南 - 通过与 AI 结对编程，将想法变为现实的终极工作站。

## 核心理念
规划就是一切。谨慎让 AI 自主规划，否则代码库会变成无法管理的乱麻。

## 工作流程
1. 用户输入需求 → 使用"智能需求理解与研发导航引擎"分析
2. 生成项目上下文文档 → 使用"项目上下文文档生成"
3. 制定实施计划 → 使用"AI 项目计划生成系统"
4. 执行代码开发 → 遵循"高质量代码开发专家"规范
5. 测试与验收 → 运行测试，检查质量清单

## 提示词使用规范

### 智能需求理解与研发导航
当用户输入任何主题、问题或需求时，按以下结构输出：
- 需求理解与意图识别（显性需求、隐性需求、背后意图）
- 关键词与概念解释
- 技术路径与开源项目
- 专家范式与高层洞见

### 项目上下文文档生成
包含：项目概要、范围定义、关键实体与关系、功能模块拆解、技术方向与关键决策、交互风格约定、当前进展总结、后续计划与风险。

### AI 项目计划生成系统
- 只生成计划文档，不执行任何代码实现
- 每个层级的计划文档都需至少输出一项可视化视图
- 使用 Mermaid、Markdown 表格或思维导图语法
- 文件命名规范：`plan_XX_任务名.md`

## 文件位置规则
- Python 源代码：`/src`，模块名小写，下划线分隔
- 测试代码：`/tests`，命名 `test_模块名.py`
- 文档：`/docs`，命名 `模块名_说明.md`
- 临时输出：`/output`，自动生成时间戳后缀
- 配置文件：`/configs`

## 代码风格规范
- Python：严格遵守 PEP8，函数名用小写加下划线；类名大驼峰；常量全大写
- JavaScript/TypeScript：使用 ESLint 规范，函数名用小驼峰；类名大驼峰；常量全大写
- 导入顺序：标准库 → 第三方库 → 项目内模块

## 质量检查清单
- 代码逻辑正确，功能完整
- 命名规范，注释清晰
- 错误处理完善
- 性能表现良好
- 安全漏洞排查
- 测试用例覆盖
- 文档完整准确
- 代码风格统一

## 禁止操作
- 禁止在项目根目录生成任何非文档规范文件
- 禁止修改 `i18n/` 目录下的原始提示词文件
- 禁止修改 `vibe-coding-cn-main/` 下的原始文件
- 所有新文件必须放入 `trae-rules/` 或导出目录

## 文档更新流程
1. 长对话或复杂项目后 → 生成项目上下文文档
2. 新功能开发前 → 更新项目上下文文档
3. 跨会话迁移时 → 导入项目上下文文档
